# MaaS Deployment Architecture: Intended vs Current Implementation

## Your Understanding is CORRECT ✅

> "IIUC from Platform perspective MaaS should be 'hidden' behind the AI Gateway operator. 
> So AI Gateway operator should deploy these manifests, not Platform operator directly.
> DSC passes config to AI Gateway Module CR, which deploys MaaS accordingly, 
> again injecting MaaS specific configuration to MaaS CR."

**This is the INTENDED architecture.** However, it is **NOT yet implemented**.

---

## The Intended Architecture (Your Description)

```
User creates DSC
  ↓
DSC.spec.components.aigateway.modelsasservice.managementState = Managed
  ↓
Platform Operator (opendatahub-operator) creates AIGateway Module CR
  ↓
Module Framework deploys ai-gateway-operator
  ↓
ai-gateway-operator watches AIGateway CR
  ↓
AIGateway.spec.modelsAsService contains MaaS configuration
  ↓
ai-gateway-operator deploys maas-controller with injected config
  ↓
maas-controller is "hidden" behind ai-gateway-operator
```

**Platform operator does NOT deploy maas-controller directly** - it's delegated to ai-gateway-operator.

## The Critical Gap

### The Config Flow is Broken

1. **Platform side (PR #3665):**
   - Can accept: `DSC.spec.components.aigateway.modelsasservice`
   - Creates: `AIGateway` CR (module CR managed by platform)
   - Problem: Platform's AIGateway type has `modelsAsService` field, but...

2. **AI Gateway Operator side (current):**
   - AIGateway CRD spec does NOT have `modelsAsService` field
   - Result: Platform cannot pass config through because the field doesn't exist in the actual CR!

3. **Current workaround:**
   - Component handler still deploys maas-controller directly
   - Reads config from: `DSC.spec.components.kserve.modelsAsService` (old location)
   - Also reads from: `DSC.spec.components.aigateway.modelsAsService` (new location, PR #3665)
   - Bypasses the whole "hidden behind ai-gateway-operator" architecture

## What Needs to Happen for the Intended Architecture

### Phase 1: ai-gateway-operator gains MaaS support

**Changes needed in `ai-gateway-operator`:**

1. **Update AIGateway CRD:**
   ```go
   type AIGatewaySpec struct {
       BatchGateway    BatchGatewayComponent    `json:"batchGateway,omitempty"`
       ModelsAsService ModelsAsServiceComponent `json:"modelsAsService,omitempty"`  // ADD THIS
   }
   
   type ModelsAsServiceComponent struct {
       ManagementState string `json:"managementState,omitempty"`
       // Other MaaS-specific config fields
   }
   ```

2. **Add deployment logic in reconciler:**
   ```go
   // In initialize() function (around line 100)
   if obj.Spec.ModelsAsService.ManagementState == managedState {
       rr.Manifests = append(rr.Manifests, m.maasManifestInfo)
       // Apply params, inject config, etc.
   }
   ```

3. **Bundle maas-controller manifests:**
   - Copy or reference `models-as-a-service/deployment/overlays/odh`
   - Make available to ai-gateway-operator at runtime

4. **Handle APPLICATIONS_NAMESPACE injection:**
   - ai-gateway-operator runs in the applications namespace
   - When deploying maas-controller, inject `APPLICATIONS_NAMESPACE` env var
   - This is what PR #980 is preparing for

### Phase 2: Platform operator stops deploying MaaS

**Changes needed in `opendatahub-operator`:**

1. **Remove (or disable) component handler:**
   - `internal/controller/components/modelsasservice/` directory
   - Or make it check: "if aigateway.modelsasservice is set, skip component deployment"

2. **Let module framework handle it:**
   - Platform creates AIGateway CR with modelsAsService config
   - ai-gateway-operator reconciles and deploys maas-controller
   - Platform just watches AIGateway CR status

## Where Does PR #980 Fit?

### PR #980's Intent:

Prepare maas-controller to receive `APPLICATIONS_NAMESPACE` from the platform via:
1. Module framework injection (future: when ai-gateway-operator deploys it)
2. Environment variable resolution with fallback chain

### The Problem:

**PR #980 is being written for Phase 1 (ai-gateway-operator deployment)**
**But Phase 1 doesn't exist yet!**

Currently:
- Component handler deploys maas-controller
- Component handler does NOT use module framework's `injectModuleEnv`
- Component handler uses kustomize with param substitution only
- So `APPLICATIONS_NAMESPACE` injection won't happen

### The Timing Issue:

**Scenario A: Merge PR #980 before ai-gateway-operator has MaaS support**
- Component handler deploys maas-controller
- No `APPLICATIONS_NAMESPACE` injection happens (component handler doesn't do it)
- Fallback to `MAAS_API_NAMESPACE` from downward API (existing mechanism)
- Go code tries to read `APPLICATIONS_NAMESPACE`, doesn't find it, falls back
- **Works, but the new code path is unused**

**Scenario B: Merge PR #980 after ai-gateway-operator has MaaS support**
- ai-gateway-operator deploys maas-controller
- Module framework injects `APPLICATIONS_NAMESPACE`
- BUT: PR #980 pre-declares the env var with `valueFrom.fieldRef`
- Injection fails (can't have both `valueFrom` and `value`)
- **Breaks!**

- "The current approach breaks the future architecture"

> as part of our multi-tenancy transition."

This is a **separate concern** about where maas-api Deployments live (tenant namespaces).
But it highlights confusion about what `APPLICATIONS_NAMESPACE` represents.
