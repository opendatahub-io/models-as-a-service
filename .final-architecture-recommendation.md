## The Intended Architecture (Per Module Handler Guide)

From the guide (lines 214-216):

> The platform injects an `APPLICATIONS_NAMESPACE` environment variable into the module operator's Deployment. This tells the module operator where the shared platform namespace is, which may be needed for cross-namespace resource discovery or for deploying workloads that interact with other modules.

**Key insight from guide (lines 1389-1404):**

```
Platform operator pod
  env:
    RELATED_IMAGE_ODH_MAAS_CONTROLLER_IMAGE=quay.io/...@sha256:aaa
    APPLICATIONS_NAMESPACE=opendatahub
        │
        │  injectModuleEnv
        │  ├── find Deployments in rr.Resources by DeploymentName
        │  ├── locate target container by ContainerName (default: "manager")
        │  ├── APPLICATIONS_NAMESPACE → inject as env var
        │
        ▼
Module operator pod (maas-controller in future)
  container "manager":
    env:
      APPLICATIONS_NAMESPACE=opendatahub
```

**The platform module framework automatically injects `APPLICATIONS_NAMESPACE`** via the `injectModuleEnv` action (lines 1281-1295).

**YOU MUST NOT PRE-DECLARE IT** in manager.yaml (lines 653-654):

> You do not need to declare these env vars in the manifest; the platform adds them automatically based on the handler's `ModuleConfig`.

| **Local dev** | Can use `--maas-api-namespace` flag |

The Go fallback chain already handles this:
```go
func resolveApplicationsNamespace(flagValue string) string {
    if v := strings.TrimSpace(flagValue); v != "" {
        return v  // CLI flag (dev/testing)
    }
    if v := strings.TrimSpace(os.Getenv(applicationsNamespaceEnv)); v != "" {
        return v  // Module framework injection (future)
    }
    if v := strings.TrimSpace(os.Getenv("MAAS_API_NAMESPACE")); v != "" {
        return v  // Downward API (current component handler)
    }
    return ""
}
```

## Why the Reviewer is Correct

From the Module Handler Developer Guide (lines 653-654):

> You do not need to declare these env vars in the manifest; the platform adds them automatically based on the handler's `ModuleConfig`.

And from line 1509:

> **Namespace injection (`APPLICATIONS_NAMESPACE`)** The platform always injects `APPLICATIONS_NAMESPACE` into the module operator's target container. No handler configuration is needed -- this injection is automatic for all modules.

**The reviewer (ryancham715) understands this architecture.** Their concerns:

1. ✅ **"I don't think we need to declare APPLICATIONS_NAMESPACE for the platform to successfully inject it"**
   - Correct! Module framework's `injectModuleEnv` does this automatically
   - Pre-declaring breaks the injection (can't have both `valueFrom` and `value`)

2. ✅ **"Setting valueFrom will actually cause issues to that process"**
   - 100% correct! The `injectModuleEnv` code (guide lines 1389-1404) only sets `value`
   - Having both `valueFrom` and `value` is invalid Kubernetes

3. ⚠️ **"maas-api would be deployed to other namespaces than opendatahub"**
   - This is about where maas-api Deployments live (tenant namespaces)
   - Separate concern but highlights confusion about what the env var represents
