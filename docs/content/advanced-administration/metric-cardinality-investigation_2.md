# [SPIKE] High Cardinality Per-User Metrics in Prometheus

## Executive Summary

MaaS tracks token consumption, request counts, and rate-limit violations **per user** via Limitador metrics at the gateway layer. Each unique combination of label values creates a separate time series in Prometheus. As the user base grows, this creates a cardinality challenge.

**Finding**: The current design is sustainable up to ~50k concurrent users on a single Prometheus instance with default OpenShift settings. Beyond that, architectural changes (remote storage or metric aggregation) are required. The tipping point depends on label cardinality, retention period, and query patterns.

## Current Metric Architecture

### Where per-user metrics come from

Per-user metrics are **not generated by maas-api**. They are generated by **Limitador** (Kuadrant's rate-limiting engine) at the Istio gateway, with labels injected by the **wasm-shim** based on auth context and response body data.

```
User request
  → Istio Gateway
    → Authorino (authentication, identity enrichment)
      → Limitador (rate limiting, counter increment, metric emission)
        → Model Server (inference)
          → Response body (usage.total_tokens, model name)
```

The TelemetryPolicy (`deployment/base/observability/gateway-telemetry-policy.yaml`) defines which labels are added:

```yaml
spec:
  metrics:
    default:
      labels:
        model: responseBodyJSON("/model")
        tier: auth.identity.tier
        user: auth.identity.userid
        subscription: auth.identity.selected_subscription
        organization_id: auth.identity.organizationId
        cost_center: auth.identity.costCenter
```

### Current per-user metrics

| Metric | Type | Labels | Purpose |
|--------|------|--------|---------|
| `authorized_hits` | Counter | `user`, `tier`, `model`, `subscription`, `organization_id`, `cost_center`, `limitador_namespace` | Token consumption (from `usage.total_tokens` in response) |
| `authorized_calls` | Counter | `user`, `tier`, `subscription`, `organization_id`, `cost_center`, `limitador_namespace` | Requests allowed (not rate-limited) |
| `limited_calls` | Counter | `user`, `tier`, `subscription`, `organization_id`, `cost_center`, `limitador_namespace` | Requests denied by rate limits |

### Metrics that intentionally avoid per-user labels

| Metric | Labels | Rationale |
|--------|--------|-----------|
| `istio_request_duration_milliseconds_bucket` | `tier` only (no `user`) | Histograms have high base cardinality (default 23 buckets per series). Adding `user` would multiply by user count. |
| `vllm:*` (all vLLM metrics) | `model_name` only | vLLM has no visibility into user identity; metrics are per-model. |

This was a deliberate design decision documented in the observability guide: *"Per-user latency is not exposed on the gateway histogram to keep cardinality bounded."*

## Cardinality Analysis

### Formula

Total time series for Limitador metrics:

```
series = metrics × users × tiers × models × subscriptions × namespaces
```

For `authorized_hits` (the highest-cardinality metric because it includes `model`):

```
series_per_metric = users × tiers × models × subscriptions × limitador_namespaces
```

For `authorized_calls` and `limited_calls` (no `model` label):

```
series_per_metric = users × tiers × subscriptions × limitador_namespaces
```

### Realistic label value counts

| Label | Typical range | Notes |
|-------|---------------|-------|
| `user` | 10 - 1M | The high-cardinality dimension. Each unique user ID creates new series. |
| `tier` | 3-5 | free, premium, enterprise (bounded, defined by platform admin) |
| `model` | 5-50 | Number of deployed LLM models (bounded, defined by platform admin) |
| `subscription` | 5-50 | Number of MaaSSubscription CRs (bounded, admin-managed) |
| `organization_id` | 1-20 | Organization count (bounded) |
| `cost_center` | 1-50 | Cost centers (bounded) |
| `limitador_namespace` | 1-5 | Typically 1 per Kuadrant deployment |

**Key insight**: `user` is the only unbounded dimension. All other labels are admin-controlled and grow slowly.

### Cardinality projections

Assumptions: 3 tiers, 10 models, 10 subscriptions, 1 limitador_namespace. Not every user hits every model/tier/subscription combination, so we use an **effective multiplier** of ~5 unique label combinations per active user (empirical estimate for typical usage patterns).

| Users | `authorized_hits` series | `authorized_calls` series | `limited_calls` series | **Total series** | Impact |
|-------|--------------------------|---------------------------|------------------------|------------------|--------|
| 100 | ~500 | ~150 | ~50 | **~700** | Negligible |
| 1,000 | ~5,000 | ~1,500 | ~500 | **~7,000** | Negligible |
| 10,000 | ~50,000 | ~15,000 | ~5,000 | **~70,000** | Comfortable |
| 50,000 | ~250,000 | ~75,000 | ~25,000 | **~350,000** | Approaching limits |
| 100,000 | ~500,000 | ~150,000 | ~50,000 | **~700,000** | Needs tuning |
| 1,000,000 | ~5,000,000 | ~1,500,000 | ~500,000 | **~7,000,000** | Exceeds single Prometheus |

**Note**: These are **active series** (users who made at least one request during the retention period). Inactive users don't create series. The real question is concurrent active users, not total registered users.

### Prometheus capacity limits

| Configuration | Active series limit | Notes |
|---------------|---------------------|-------|
| OpenShift user-workload-monitoring (default) | ~1M active series | Default resource limits; shared with all user workloads |
| OpenShift user-workload-monitoring (tuned) | ~5M active series | Increased memory (16-32GB), dedicated PV for TSDB |
| Standalone Prometheus | ~10M active series | Dedicated instance with 64GB RAM, NVMe storage |
| Mimir / Thanos / VictoriaMetrics | 100M+ active series | Horizontally scalable, remote storage |

### Tipping points

| User scale | Status | Action required |
|------------|--------|-----------------|
| **< 10k users** | Safe | No action needed. Default OpenShift monitoring handles this. |
| **10k - 50k users** | Monitor | Set up cardinality alerts. Tune Prometheus resource limits. |
| **50k - 100k users** | Warning | Tune Prometheus, increase retention PV. Consider metric aggregation via recording rules. |
| **100k+ users** | Critical | Migrate to remote storage (Mimir, Thanos, or VictoriaMetrics). Or redesign metrics. |

## Performance Impact Details

### 1. Query Performance

High cardinality primarily affects queries that aggregate across the `user` dimension:

```promql
# SLOW at high cardinality — scans all user series
sum by (tier) (rate(authorized_hits[5m]))

# FAST — already scoped to one user
sum(authorized_hits{user="alice"})

# VERY SLOW — sorts all user series
topk(10, sum by (user) (authorized_hits))
```

At 100k users, a `topk` query across all users can take 10-30 seconds. Dashboard panels using these queries will timeout or cause Prometheus OOM.

### 2. Storage Costs

Each active time series consumes approximately:
- **Ingestion**: ~1-2 bytes per sample (compressed TSDB)
- **Samples per day**: 2,880 (at 30s scrape interval)
- **Per series per day**: ~3-6 KB

| Users | Active series | Storage per day | Storage per 15 days (default retention) |
|-------|---------------|-----------------|----------------------------------------|
| 10k | ~70k | ~350 MB | ~5 GB |
| 100k | ~700k | ~3.5 GB | ~50 GB |
| 1M | ~7M | ~35 GB | ~500 GB |

### 3. Ingestion Load

Limitador exposes all counters on its `/metrics` endpoint, scraped every 30 seconds. At high cardinality:

- **Scrape size**: Each series adds ~100 bytes to the scrape response
- **100k users**: ~70 MB per scrape (700k series × 100 bytes)
- **Scrape duration**: May exceed the 30s interval, causing missed scrapes

## Alternative Metric Designs

### Option 1: Recording rules for pre-aggregation (Recommended for 10k-100k users)

Keep per-user metrics for billing queries but pre-aggregate for dashboards:

```yaml
groups:
- name: maas-aggregated
  interval: 1m
  rules:
  # Pre-aggregate by tier (dashboards use this instead of raw per-user data)
  - record: maas:authorized_hits:rate5m_by_tier
    expr: sum by (tier) (rate(authorized_hits[5m]))

  - record: maas:authorized_calls:rate5m_by_tier
    expr: sum by (tier) (rate(authorized_calls[5m]))

  # Pre-aggregate by model
  - record: maas:authorized_hits:rate5m_by_model
    expr: sum by (model) (rate(authorized_hits[5m]))

  # Top users (computed once per minute, not per dashboard load)
  - record: maas:authorized_hits:top20_users
    expr: topk(20, sum by (user) (rate(authorized_hits[5m])))
```

**Pros**: No architecture change; dashboards query recording rules instead of raw metrics. Per-user data remains available for billing.
**Cons**: Does not reduce storage or ingestion; only improves query performance.

### Option 2: Metric relabeling to drop user label from non-billing metrics (Recommended for 50k+ users)

Use Prometheus `metric_relabel_configs` to keep `user` only on the billing metric (`authorized_hits`) and drop it from `authorized_calls` and `limited_calls`:

```yaml
# In the Limitador ServiceMonitor
spec:
  endpoints:
  - port: http
    path: /metrics
    interval: 30s
    metricRelabelings:
    # Drop user label from authorized_calls and limited_calls
    # (keep it only on authorized_hits for billing)
    - sourceLabels: [__name__]
      regex: '(authorized_calls|limited_calls)'
      action: keep
      targetLabel: __tmp_drop_user
    - sourceLabels: [__tmp_drop_user]
      regex: '.+'
      action: labeldrop
      targetLabel: user
```

**Pros**: Reduces series count by ~2/3 (two of three metrics lose the user dimension).
**Cons**: Loses per-user request count and rate-limit tracking; only per-user token billing remains.

### Option 3: Bucket aggregation — replace per-user with per-cohort (For 100k+ users)

Instead of `user="alice"`, use a hashed bucket: `user_bucket="bucket_042"` (e.g., 1000 buckets). Exact per-user billing data moves to a dedicated billing system (PostgreSQL or a billing API).

```
# TelemetryPolicy change
labels:
  user_bucket: 'auth.identity.userid.hash() % 1000'  # Pseudocode
  tier: auth.identity.tier
  model: responseBodyJSON("/model")
```

**Pros**: Constant cardinality regardless of user count (1000 buckets × 3 tiers × 10 models = 30k series).
**Cons**: Loses per-user granularity in Prometheus. Requires a separate billing data pipeline.

### Option 4: Remote storage with downsampling (For 100k+ users)

Replace single Prometheus with a horizontally scalable solution:

| Solution | Downsampling | Multi-tenancy | OpenShift integration |
|----------|-------------|---------------|----------------------|
| **Thanos** | Yes (compactor) | Yes (receiver) | Good (RHOBS uses it) |
| **Mimir** | Yes (compactor) | Yes (native) | Good |
| **VictoriaMetrics** | Yes (vmagent) | Yes (cluster mode) | Fair |

**Pros**: Handles millions of series; no metric design changes needed.
**Cons**: Significant infrastructure cost and operational complexity.

## Recommendation

| User scale | Recommended approach |
|------------|---------------------|
| **Current (< 10k)** | **No changes needed.** Monitor cardinality with `prometheus_tsdb_head_series`. |
| **Near-term (10k-50k)** | **Option 1**: Add recording rules for dashboard queries. Keep per-user data for billing. Add cardinality alerting. |
| **Medium-term (50k-100k)** | **Options 1 + 2**: Recording rules + drop `user` from `authorized_calls`/`limited_calls`. Tune Prometheus resources. |
| **Long-term (100k+)** | **Option 4**: Migrate to remote storage (Thanos/Mimir). Or **Option 3** if a separate billing pipeline is acceptable. |

### Immediate action items

1. **Add a cardinality monitoring alert**:
    ```yaml
    - alert: MaaSHighMetricCardinality
      expr: count({__name__=~"authorized_hits|authorized_calls|limited_calls"}) > 100000
      for: 10m
      labels:
        severity: warning
      annotations:
        summary: "MaaS Limitador metrics cardinality exceeds 100k series"
    ```

2. **Add recording rules** for the Platform Admin Grafana dashboard so it queries pre-aggregated data instead of scanning all user series.

3. **Document the cardinality budget** in the observability guide: per-user Limitador metrics are expected to be the dominant cardinality source; all other MaaS metrics (vLLM, Istio, Authorino) are bounded.

## Appendix: What about the `subscription`, `organization_id`, and `cost_center` labels?

The TelemetryPolicy recently added three additional labels to all Limitador metrics:

```yaml
subscription: auth.identity.selected_subscription
organization_id: auth.identity.organizationId
cost_center: auth.identity.costCenter
```

These are low-cardinality individually (admin-managed, typically < 50 values each). However, they **multiply with `user`** since each user may have a different subscription/org/cost-center combination. In the worst case, this increases the effective multiplier per user from ~5 to ~10-15 unique label combinations.

**Recommendation**: Evaluate whether `organization_id` and `cost_center` are needed on every metric, or only on `authorized_hits` (for billing). Dropping them from `authorized_calls`/`limited_calls` via metric relabeling would reduce cardinality with minimal information loss, since these counters are primarily used for capacity planning (by tier), not billing (by org).
