# [SPIKE] High Cardinality Per-User Metrics in Prometheus

## Executive Summary

MaaS tracks token consumption, request counts, and rate-limit violations **per user** via Limitador metrics at the gateway layer. Each unique combination of label values creates a separate time series in Prometheus. As the user base grows, this creates a cardinality challenge.

**Finding**: The current design faces cardinality risks even at moderate scale. Several labels (`model`, `subscription`, `organization_id`, `cost_center`) are effectively unbounded unless hard limits are enforced. A worst-case analysis with realistic label ranges produces ~2.4M time series *before* the `user` dimension is even considered. The ~50k user threshold below is an **analytical estimate, not validated by load testing**, and depends heavily on assumptions about label combination frequency that need vetting with real customer data.

## Current Metric Architecture

### Where per-user metrics come from

Per-user metrics are **not generated by maas-api**. They are generated by **Limitador** (Kuadrant's rate-limiting engine) at the Istio gateway, with labels injected by the **wasm-shim** based on auth context and response body data.

```
User request
  -> Istio Gateway
    -> Authorino (authentication, identity enrichment)
      -> Limitador (rate limiting, counter increment, metric emission)
        -> Model Server (inference)
          -> Response body (usage.total_tokens, model name)
```

The TelemetryPolicy (`deployment/base/observability/gateway-telemetry-policy.yaml`) defines which labels are added:

```yaml
spec:
  metrics:
    default:
      labels:
        model: responseBodyJSON("/model")
        tier: auth.identity.tier
        user: auth.identity.userid
        subscription: auth.identity.selected_subscription
        organization_id: auth.identity.organizationId
        cost_center: auth.identity.costCenter
```

### Current per-user metrics

| Metric | Type | Labels | Purpose |
|--------|------|--------|---------|
| `authorized_hits` | Counter | `user`, `tier`, `model`, `subscription`, `organization_id`, `cost_center`, `limitador_namespace` | Token consumption (from `usage.total_tokens` in response) |
| `authorized_calls` | Counter | `user`, `tier`, `subscription`, `organization_id`, `cost_center`, `limitador_namespace` | Requests allowed (not rate-limited) |
| `limited_calls` | Counter | `user`, `tier`, `subscription`, `organization_id`, `cost_center`, `limitador_namespace` | Requests denied by rate limits |

### Metrics that intentionally avoid per-user labels

| Metric | Labels | Rationale |
|--------|--------|-----------|
| `istio_request_duration_milliseconds_bucket` | `tier` only (no `user`) | Histograms have high base cardinality (default 23 buckets per series). Adding `user` would multiply by user count. |
| `vllm:*` (all vLLM metrics) | `model_name` only | vLLM has no visibility into user identity; metrics are per-model. |

This was a deliberate design decision documented in the observability guide: *"Per-user latency is not exposed on the gateway histogram to keep cardinality bounded."*

## Cardinality Analysis

### Formula

Total time series for Limitador metrics:

```
series = metrics x users x tiers x models x subscriptions x org_ids x cost_centers x namespaces
```

For `authorized_hits` (the highest-cardinality metric because it includes `model`):

```
series_per_metric = users x tiers x models x subscriptions x org_ids x cost_centers x limitador_namespaces
```

For `authorized_calls` and `limited_calls` (no `model` label):

```
series_per_metric = users x tiers x subscriptions x org_ids x cost_centers x limitador_namespaces
```

### Label value counts and boundedness

| Label | Typical range | Bounded? | Notes |
|-------|---------------|----------|-------|
| `user` | 10 - 1M | **No** | The primary high-cardinality dimension. Each unique user ID creates new series. |
| `tier` | 3-5 | Yes | free, premium, enterprise (defined by platform admin) |
| `model` | 5-200+ | **No** | Number of deployed models. RHAISTRAT-730 uses 200 as "medium-large". OpenRouter exposes 653. A single API call creates the time series for the retention period. |
| `subscription` | 5-50+ | **No** | Number of MaaSSubscription CRs. No enforced upper limit. |
| `organization_id` | 1-20+ | **No** | Organization count. No enforced upper limit. |
| `cost_center` | 1-50+ | **No** | Cost centers. No enforced upper limit. |
| `limitador_namespace` | 1-5 | Yes | Typically 1 per Kuadrant deployment |

**Key concern**: Only `tier` and `limitador_namespace` are truly bounded. The other labels (`model`, `subscription`, `organization_id`, `cost_center`) are effectively unbounded unless MaaS enforces hard limits on unique value counts. The `user` dimension compounds this.

### Worst-case cardinality without the user dimension

Using average values from realistic ranges (4 tiers, 27.5 models, 27.5 subscriptions, 10.5 org_ids, 25.5 cost_centers, 3 namespaces):

```
4 x 27.5 x 27.5 x 10.5 x 25.5 x 3 = ~2,430,000 time series
```

This already exceeds comfortable Prometheus limits **before adding the `user` dimension**. Even if most combinations are sparse, the theoretical cardinality space is enormous.

### Cardinality projections

**Conservative scenario** (small deployment): 3 tiers, 10 models, 5 subscriptions, 2 org_ids, 3 cost_centers, 1 namespace.

Assuming an effective multiplier of ~5 unique label combinations per active user:

> **Note**: The ~5 combinations/user assumption is an unvalidated estimate. Real-world usage patterns may be significantly higher. A user who experiments with 15+ models (e.g., 8 Gemini + 7 Claude variants) across multiple subscriptions could easily generate 20-30+ unique label combinations. This assumption needs vetting with real customer data.

| Users | Estimated total series | Impact |
|-------|----------------------|--------|
| 100 | ~700 | Negligible |
| 1,000 | ~7,000 | Negligible |
| 10,000 | ~70,000 | Comfortable |
| 50,000 | ~350,000 | Approaching limits |
| 100,000 | ~700,000 | Needs tuning |

**Realistic scenario** (medium-large deployment per RHAISTRAT-730): 4 tiers, 200 models, 25 subscriptions, 10 org_ids, 20 cost_centers, 3 namespaces.

With ~15 unique label combinations per active user:

| Users | Estimated total series | Impact |
|-------|----------------------|--------|
| 100 | ~4,500 | Negligible |
| 1,000 | ~45,000 | Comfortable |
| 10,000 | ~450,000 | Warning |
| 50,000 | ~2,250,000 | Exceeds defaults |
| 100,000 | ~4,500,000 | Exceeds single Prometheus |

### Prometheus capacity limits

| Configuration | Active series limit | Notes |
|---------------|---------------------|-------|
| OpenShift user-workload-monitoring (default) | ~1M active series | Default resource limits; shared with all user workloads |
| OpenShift user-workload-monitoring (tuned) | ~5M active series | Increased memory (16-32GB), dedicated PV for TSDB |
| Standalone Prometheus | ~10M active series | Dedicated instance with 64GB RAM, NVMe storage |
| Mimir / Thanos / VictoriaMetrics | 100M+ active series | Horizontally scalable, remote storage |

### Tipping points

| User scale | Status | Action required |
|------------|--------|-----------------|
| **< 1k users** (small deployment) | Safe | No action needed with conservative label counts. |
| **1k - 10k users** | Monitor | Set up cardinality alerts. Assess actual label combination counts. |
| **10k - 50k users** | Warning | Tune Prometheus resources. Strongly consider dropping high-cardinality labels. |
| **50k+ users** | Critical | Must reduce label cardinality or migrate to remote storage. |

## Performance Impact Details

### 1. Impact on Limitador itself

Limitador exposes all counters on its `/metrics` endpoint. As cardinality grows, the `/metrics` response body grows proportionally. This affects Limitador's own performance:

- **Memory**: Each counter maintained in memory increases Limitador's resident set size.
- **Serialization**: Every scrape requires Limitador to serialize all counters to Prometheus text format. At 700k+ series, this serialization can consume significant CPU.
- **Scrape duration**: Large `/metrics` responses increase scrape time. If scrape duration approaches the scrape interval (30s), Prometheus will miss scrapes and data gaps appear.
- **Network**: At 100k+ series, each scrape response can be tens of megabytes, adding non-trivial network load on every 30s interval.

The impact on Limitador's core rate-limiting function (latency of rate-limit decisions) needs investigation - it is unclear whether counter growth degrades the hot path.

### 2. Query Performance

High cardinality primarily affects queries that aggregate across the `user` dimension:

```promql
# SLOW at high cardinality - scans all user series
sum by (tier) (rate(authorized_hits[5m]))

# FAST - already scoped to one user
sum(authorized_hits{user="alice"})

# VERY SLOW - sorts all user series
topk(10, sum by (user) (authorized_hits))
```

At 100k users, a `topk` query across all users can take 10-30 seconds. Dashboard panels using these queries will timeout or cause Prometheus OOM.

### 3. Storage Costs

Each active time series consumes approximately:
- **Ingestion**: ~1-2 bytes per sample (compressed TSDB)
- **Samples per day**: 2,880 (at 30s scrape interval)
- **Per series per day**: ~3-6 KB

| Users | Active series (realistic) | Storage per day | Storage per 15 days (default retention) |
|-------|--------------------------|-----------------|----------------------------------------|
| 10k | ~450k | ~2.3 GB | ~34 GB |
| 50k | ~2.25M | ~11.3 GB | ~170 GB |
| 100k | ~4.5M | ~22.5 GB | ~340 GB |

### 4. Ingestion Load

Limitador exposes all counters on its `/metrics` endpoint, scraped every 30 seconds. At high cardinality:

- **Scrape size**: Each series adds ~100 bytes to the scrape response
- **50k users (realistic)**: ~225 MB per scrape (2.25M series x 100 bytes)
- **Scrape duration**: Will likely exceed the 30s interval at this scale, causing missed scrapes and data gaps

## Realistic Options

Based on review, the following options are viable:

### Option 1: Drop high-cardinality labels by default (Recommended)

Remove the `user` label (and potentially `organization_id`, `cost_center`) from Limitador metrics by default. Offer a `captureUser: true` opt-in for customers who need per-user metrics and understand the cardinality implications.

This aligns with the `captureUser: false` proposal. Per-user billing data would move to a dedicated system (e.g., PostgreSQL via maas-api) rather than being derived from Prometheus metrics.

**Pros**: Immediately bounds cardinality. Simple to implement via TelemetryPolicy configuration.
**Cons**: Loses per-user Prometheus queries. Requires a separate billing data pipeline.

### Option 2: Move high-cardinality labels to logs

Instead of emitting per-user data as Prometheus metric labels, emit them as structured log entries. Logs handle high cardinality naturally (each entry is independent, no combinatorial explosion).

```
# Instead of metric labels:
authorized_hits{user="alice", model="granite-3-8b", subscription="team-alpha", ...} 42

# Emit structured log:
{"event":"token_usage", "user":"alice", "model":"granite-3-8b", "subscription":"team-alpha", "tokens":42, "timestamp":"..."}
```

Logs can be queried via Loki/Elasticsearch for billing and auditing. Prometheus metrics retain only the bounded labels (`tier`, `model`, `limitador_namespace`) for dashboards.

**Pros**: Per-user data preserved for billing/auditing. Prometheus cardinality stays bounded. Log systems handle high cardinality well.
**Cons**: Requires log aggregation infrastructure. Billing queries move from PromQL to LogQL/SQL.

### Option 3: Enforce hard limits on unique label values

Enforce maximum unique value counts for each label in MaaS so that worst-case series count is predictable:

| Label | Proposed limit | Rationale |
|-------|---------------|-----------|
| `model` | 50 | Platform admin controls deployments |
| `subscription` | 100 | Admin-managed |
| `organization_id` | 20 | Tenant count |
| `cost_center` | 50 | Organizational structure |

With these limits and `user` dropped, worst-case series = 3 metrics x 5 tiers x 50 models x 100 subs x 20 orgs x 50 cost_centers x 3 namespaces = ~4.5B. This is still too large - hard limits alone don't solve the combinatorial problem. **Hard limits only work if combined with dropping some labels.**

**Pros**: Makes cardinality predictable. Prevents runaway growth.
**Cons**: Limits alone don't solve the combinatorial explosion. Must be combined with other approaches.

### Option 4: Recording rules for pre-aggregation (Query optimization only)

Keep per-user metrics but pre-aggregate for dashboards:

```yaml
groups:
- name: maas-aggregated
  interval: 1m
  rules:
  - record: maas:authorized_hits:rate5m_by_tier
    expr: sum by (tier) (rate(authorized_hits[5m]))
  - record: maas:authorized_hits:rate5m_by_model
    expr: sum by (model) (rate(authorized_hits[5m]))
  - record: maas:authorized_hits:top20_users
    expr: topk(20, sum by (user) (rate(authorized_hits[5m])))
```

**Pros**: No architecture change; dashboards query recording rules instead of raw metrics.
**Cons**: Does **not** reduce storage or ingestion load. Only improves query performance. This is a partial optimization, not a solution to the cardinality problem.

## Recommendation

| Approach | When to use |
|----------|-------------|
| **Option 1 (Drop labels by default)** | Default for all deployments. Safest path. |
| **Option 2 (Labels to logs)** | When per-user data is needed for billing/auditing but not real-time dashboards. |
| **Option 4 (Recording rules)** | As a complement to Option 1/2 for dashboard query optimization. |
| **Option 3 (Hard limits)** | As defense-in-depth alongside Option 1 or 2, not standalone. |

### Immediate action items

1. **Validate assumptions with real customer data**: Reach out to customers willing to share figures on:
   - Number of deployed models
   - Number of active subscriptions and organizations
   - Number of concurrent active users
   - Typical usage patterns (how many models does a single user touch?)

2. **Add a cardinality monitoring alert**:
    ```yaml
    - alert: MaaSHighMetricCardinality
      expr: count({__name__=~"authorized_hits|authorized_calls|limited_calls"}) > 100000
      for: 10m
      labels:
        severity: warning
      annotations:
        summary: "MaaS Limitador metrics cardinality exceeds 100k series"
    ```

3. **Investigate Limitador impact**: Determine whether growing counter counts affect Limitador's rate-limiting hot path (latency of rate-limit decisions), not just scrape performance.

4. **Evaluate `captureUser: false` as default**: Make per-user metric capture opt-in rather than opt-out, with clear documentation on the cardinality implications of enabling it.

5. **Document the cardinality budget** in the observability guide: per-user Limitador metrics are the dominant cardinality source; all other MaaS metrics (vLLM, Istio, Authorino) are bounded.

## Appendix: What about the `subscription`, `organization_id`, and `cost_center` labels?

The TelemetryPolicy recently added three additional labels to all Limitador metrics:

```yaml
subscription: auth.identity.selected_subscription
organization_id: auth.identity.organizationId
cost_center: auth.identity.costCenter
```

These labels are **not bounded** unless MaaS enforces limits on unique values. They multiply with every other dimension: each unique subscription/org/cost-center combination creates new series for every user-tier-model combination.

Even without the `user` dimension, the cross-product of these labels with `model` creates significant cardinality. Using average values from realistic ranges:

```
4 tiers x 27.5 models x 27.5 subscriptions x 10.5 orgs x 25.5 cost_centers x 3 namespaces
= ~2,430,000 time series (per metric, without user)
```

**Recommendation**: These labels should be evaluated for whether they belong on Prometheus metrics at all, or whether they are better suited as structured log attributes (Option 2). If kept on metrics, they should only appear on the billing metric (`authorized_hits`), not on `authorized_calls`/`limited_calls` which are primarily used for capacity planning by tier.
