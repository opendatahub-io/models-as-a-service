# Live Cluster Test Plan: APPLICATIONS_NAMESPACE Support

## Goal
Verify that maas-controller correctly resolves the applications namespace using the 3-tier fallback mechanism.

## Test Scenarios

### Scenario 1: Component Handler Deployment (Current ODH Path)
**Setup:**
- Deploy via ODH operator component handler
- Component handler deploys maas-controller using kustomize
- `MAAS_API_NAMESPACE` comes from downward API in manager.yaml

**Expected Behavior:**
1. maas-controller pod starts successfully
2. Logs show: namespace resolved from `MAAS_API_NAMESPACE` env var (3rd tier fallback)
3. maas-controller creates maas-api deployments in the correct namespace
4. No errors about missing APPLICATIONS_NAMESPACE

**Verification Commands:**
```bash
# Check pod is running
kubectl get pods -n opendatahub -l control-plane=maas-controller

# Check env vars in the pod
kubectl get pod -n opendatahub -l control-plane=maas-controller -o jsonpath='{.items[0].spec.containers[0].env[?(@.name=="MAAS_API_NAMESPACE")]}' | jq .
kubectl get pod -n opendatahub -l control-plane=maas-controller -o jsonpath='{.items[0].spec.containers[0].env[?(@.name=="APPLICATIONS_NAMESPACE")]}' | jq .

# Check logs for namespace resolution
kubectl logs -n opendatahub -l control-plane=maas-controller | grep -i "namespace"
```

### Scenario 2: Manual Injection Test (Future Module Framework Simulation)
**Setup:**
- Edit the deployment to manually inject APPLICATIONS_NAMESPACE as a value (simulating module framework)
- Remove or override MAAS_API_NAMESPACE

**Expected Behavior:**
1. maas-controller should prefer APPLICATIONS_NAMESPACE over MAAS_API_NAMESPACE
2. Should work even if MAAS_API_NAMESPACE is removed

**Test Commands:**
```bash
# Patch deployment to add APPLICATIONS_NAMESPACE as a value field
kubectl patch deployment -n opendatahub maas-controller --type=json -p='[
  {
    "op": "add",
    "path": "/spec/template/spec/containers/0/env/-",
    "value": {
      "name": "APPLICATIONS_NAMESPACE",
      "value": "opendatahub"
    }
  }
]'

# Wait for rollout
kubectl rollout status deployment/maas-controller -n opendatahub

# Verify it works
kubectl logs -n opendatahub -l control-plane=maas-controller | grep -i "applications.*namespace"
```

### Scenario 3: CLI Flag Override
**Setup:**
- Patch deployment to add --maas-api-namespace flag

**Expected Behavior:**
1. Flag value should win over all env vars

**Test Commands:**
```bash
# Patch to add flag
kubectl patch deployment -n opendatahub maas-controller --type=json -p='[
  {
    "op": "add",
    "path": "/spec/template/spec/containers/0/args/-",
    "value": "--maas-api-namespace=custom-test-ns"
  }
]'

# Verify flag is respected
kubectl logs -n opendatahub -l control-plane=maas-controller | tail -50
```

### Scenario 4: No Namespace Available (Error Case)
**Setup:**
- Remove all namespace sources (MAAS_API_NAMESPACE, don't set APPLICATIONS_NAMESPACE, no flag)

**Expected Behavior:**
1. Pod should fail to start
2. Error message should be clear: "applications namespace is required"

**Test Commands:**
```bash
# Patch to remove MAAS_API_NAMESPACE
kubectl patch deployment -n opendatahub maas-controller --type=json -p='[
  {
    "op": "remove",
    "path": "/spec/template/spec/containers/0/env/X"
  }
]'
# (Replace X with the index of MAAS_API_NAMESPACE)

# Check logs show clear error
kubectl logs -n opendatahub -l control-plane=maas-controller | grep -i "error\|required"
```

## Success Criteria

✅ All scenarios pass
✅ No conflicts between valueFrom and value in env vars
✅ Fallback order is correct (flag → APPLICATIONS_NAMESPACE → MAAS_API_NAMESPACE)
✅ Clear error messages when namespace cannot be resolved
✅ odh-trusted-ca-bundle volume mount is present
